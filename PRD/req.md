# 需求文档：通用任务处理中心 (Task Engine)

## 1. 项目背景与目标
**现状痛点**：目前系统中各开发人员私自开启线程池，技术标准不一，导致存在大量无效线程、任务堆积及 Direct Memory OOM（内存溢出）风险。
**核心目标**：构建一个公共的、规范化的任务处理器，强制收口全系统的异步任务，提供可视化监控、动态弹性调优及异常隔离能力。

---

## 2. 任务分类与隔离策略
为了防止不同性质的任务互相干扰（如耗时长的清理任务阻塞了高频的告警处理），系统需将任务分为四类，并进行物理隔离：

| 任务类型 | 场景描述 | 调度策略建议 | 隔离等级 |
| :--- | :--- | :--- | :--- |
| **Type 1: 初始化任务** | 系统启动时调用一次，执行完即销毁。 | 独立短线程池 | 高 |
| **Type 2: 周期性定时** | Cron 表达式触发（如数据采集）。 | ThreadPoolTaskScheduler | 中 |
| **Type 3: 高频流式任务** | 主动上报、高 QPS（达10000/s）业务逻辑。 | 固定大小高性能线程池 + 阻塞队列 | 极高 |
| **Type 4: 后台维护任务** | 定时清理日志、归档数据等低优先级任务。 | 共享后台任务池 | 低 |

---

## 3. 核心功能需求

### 3.1 统一任务注册机制
*   提供统一的接口（如 `ITaskProcessor`），所有业务逻辑必须实现该接口。
*   注册任务时强制要求提供 `TaskName`（用于线程重命名）和 `TaskPriority`。

### 3.2 智能看板与可视化监控
*   **实时指标**：每个任务的当前 QPS、平均响应时间 (RT)、成功/失败总数。
*   **积压预警**：实时监控队列深度（Waiting Queue Size），若超过阈值需记录预警日志。
*   **资源占用**：监控每个任务池的活跃线程数、峰值线程数。
*   **数据操作**：支持通过接口/UI 清除特定任务或全局的统计数据。

### 3.3 动态弹性与背压控制
*   **动态扩容**：当检测到任务积压（队列达到设定比例）且活跃线程已满时，支持动态增加核心线程数（需设定全局最大红线，防止撑爆内存）。
*   **自动缩容**：当负载下降且线程空闲超过设定时间，自动回收线程回到初始水位。
*   **拒绝策略控制**：针对不同任务提供可选策略（如告警类任务：丢弃并报警；清理类任务：由调用者执行）。

### 3.4 稳定性保障（防 OOM 专项）
*   **优雅停机**：系统关闭时，支持最长 30s 的任务补偿执行，确保数据不丢失。
*   **上下文透传**：支持异步线程间的 TraceID（全链路追踪）透传。
*   **线程命名规范**：所有线程必须按 `TaskType-TaskName-ID` 格式命名，方便定位问题。

---

## 4. 技术规格要求（给 AI 的实现指令）
1.  **框架基础**：基于 Spring Boot 环境，使用 `ThreadPoolTaskExecutor` 或 `ThreadPoolTaskScheduler` 进行封装。
2.  **并发容器**：使用 `ConcurrentHashMap` 存储任务状态，确保监控数据的线程安全。
3.  **时间控制**：定时任务需支持 `FixedRate` 和 `FixedDelay` 两种模式。
4.  **接口设计**：
    *   `register(TaskConfig)`：任务注册。
    *   `execute(TaskContext)`：任务触发。
    *   `getStats()`：获取全量统计快照。
    *   `updateConfig(DynamicConfig)`：动态调整线程参数。

---

## 5. 交互与输出示例
*   **API示例**：`GET /monitor/task/status` 返回 JSON 格式的任务运行报表。
*   **控制台日志示例**：`[Alarm-Processor-thread-1] INFO: Task executed in 12ms, Queue: 5/2000.`
