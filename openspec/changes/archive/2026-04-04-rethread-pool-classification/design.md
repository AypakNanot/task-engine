## Context

Task Engine 当前面临线程池管理混乱、任务分类不合理的问题。需要在不破坏现有 API 太远的前提下，实现线程池的统一管控和资源优化。

**约束条件**：
- Spring Boot 3.5.x
- Java 17
- 向后兼容性有限（允许 breaking change）
- 不影响正在运行的生产任务

## Goals / Non-Goals

**Goals**:
- 建立基于资源特性的任务分类体系
- 实现线程池统一配置和管理
- 禁止运行时动态创建线程池
- 提供清晰的混合任务拆分指导

**Non-Goals**:
- 不实现自动任务分析和路由（过于复杂）
- 不实现动态线程池参数调整（属于另一个变更）
- 不实现任务优先级调度（已移除 TaskPriority）

## Decisions

### Decision 1: 按资源特性分类 TaskType

**选择**：CPU_BOUND / IO_BOUND / HYBRID / SCHEDULED / BATCH

**原因**：
- 资源特性是线程池配置的核心决定因素
- CPU 密集型需要少线程、小队列
- IO 密集型需要多线程、大队列
- 分类简单，开发者容易判断

**Alternatives Considered**:
- 按业务场景分类：过于具体，难以覆盖所有场景
- 保持现有 4 类：无法反映任务本质差异

### Decision 2: 混合任务由开发者拆分

**选择**：开发者负责将混合任务拆分为纯 CPU 和纯 IO 子任务

**原因**：
- 框架无法自动识别任务内部的资源使用模式
- 强制拆分促使开发者关注任务边界和性能特征
- 子任务可独立优化和监控

**Alternatives Considered**:
- 提供 HYBRID 线程池：会导致资源争抢，降低隔离性
- 自动识别和路由：实现复杂，准确率有限

### Decision 3: 完全禁用自定义线程池

**选择**：只能使用预配置的 5 类线程池

**原因**：
- 确保全局资源可控
- 防止开发人员随意创建线程池导致资源耗尽
- 简化运维和监控

**Alternatives Considered**:
- 允许配置范围内调整：增加了复杂性和不确定性
- 提供有限的自定义选项：仍可能导致配置混乱

### Decision 4: 线程池参数默认值

| 类型 | Core | Max | Queue | 设计理念 |
|------|------|-----|-------|----------|
| CPU_BOUND | CPU 核数 | CPU*2 | 100 | 少线程，快速失败 |
| IO_BOUND | 16 | 64 | 1000 | 多线程吸收等待时间 |
| HYBRID | 8 | 16 | 500 | 平衡型配置 |
| SCHEDULED | 4 | 4 | 0 | 定时任务不排队 |
| BATCH | 2 | 4 | 10000 | 低优先级，大队列缓冲 |

## Risks / Trade-offs

### Risk 1: 现有任务迁移成本

**风险**：存量代码需要修改 TaskType 和配置

**Mitigation**:
- 提供迁移指南和脚本
- 保留原枚举值作为别名（@Deprecated）
- 分阶段迁移，先并行后移除

### Risk 2: 开发者不理解任务拆分

**风险**：开发者可能不知道如何拆分混合任务

**Mitigation**:
- 提供详细的示例代码
- 在文档中说明判断标准
- 提供代码审查清单

### Risk 3: 配置不当导致性能下降

**风险**：统一配置可能不适配所有业务场景

**Mitigation**:
- 提供基于 CPU 核数的自动计算
- 允许通过配置覆盖默认值
- 提供监控指标辅助调优

## Migration Plan

### Phase 1: 准备阶段
1. 新增 TaskType 枚举值（不删除旧的）
2. 新增配置结构（兼容旧配置）
3. 更新文档和示例

### Phase 2: 迁移阶段
1. 开发者修改代码使用新类型
2. 更新配置文件
3. 验证功能正常

### Phase 3: 清理阶段
1. 删除旧 TaskType 枚举值
2. 删除旧配置兼容逻辑
3. 清理相关测试

## Open Questions

1. **是否需要提供任务类型推荐工具**？
   - 静态分析代码，建议任务类型
   - 增加开发成本，暂不实现

2. **是否需要根据运行时指标动态调整分类**？
   - 属于进阶功能，二期考虑

3. **BATCH 类型是否需要独立的拒绝策略**？
   - 可能需要 BLOCK_WAIT 保证批处理完整性
